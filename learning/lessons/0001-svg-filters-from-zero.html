SVG Filters from Zero

◦ Start of the series Next: 0002 · How a PNG Bends Light Glossary

No SVG knowledge assumed. This lesson teaches one thing: an SVG filter is an assembly line for pixels, and you can read one. Everything in your LiquidGlass component sits on this. The demo here works in every browser — no Chromium caveats yet.

The problem it solves

CSS effects are mostly single dials: blur(4px), brightness(1.2), drop-shadow(…). One input, one canned effect, no way to combine them in interesting orders or feed one effect's output into another. Before SVG filters, anything fancier — embossing, glows built from a shape's own silhouette, textured distortion — meant baking images in Photoshop and shipping PNGs that couldn't adapt to content.

SVG filters are the escape hatch the web has had since 2001: instead of one canned effect, you get a box of small pixel machines (officially "filter primitives") and permission to wire them together however you like.

The idea: an assembly line for pixels

Three rules cover 90% of every filter you'll ever read:

1. A filter is a named list of machines. It lives in an <svg> that can be zero-sized and invisible — it's a definition, not a picture. Each machine is a tag starting with fe ("filter effect"): feGaussianBlur blurs, feOffset shifts, feColorMatrix recolors, feMerge stacks layers.

2. Machines pass pixels via in and result. A machine names its output with result="something"; the next machine picks it up with in="something". Two inputs exist for free: SourceGraphic (the element being filtered, in full color) and SourceAlpha (just its silhouette — black, with the element's transparency).

3. CSS points an element at the line with filter: url(#id). The element's own pixels go in at the start; whatever comes off the end of the line is what gets painted. 

<svg width="0" height="0">                          <!-- invisible toolbox -->
  <filter id="fancy">
    <feGaussianBlur in="SourceAlpha" stdDeviation="6" result="soft" />
    <feOffset       in="soft" dx="10" dy="10"      result="moved" />
    ...
  </filter>
</svg>

.card { filter: url(#fancy); }                      /* CSS connects the element */

Read it like a recipe, top to bottom: take the silhouette, soften it, move it down-right… That's the whole literacy skill. (Full primitive vocabulary: MDN: <filter>.)

Boundaries

  • filter: url() works everywhere — Chrome, Safari, Firefox. The compatibility minefield only starts when the same filters are used in backdrop-filter (lesson 0002).
  • For simple cases, don't use this. A plain shadow should be CSS filter: drop-shadow(…) or box-shadow — one line, faster. Reach for an SVG filter line only when machines must be composed in custom ways.
  • One machine we deliberately skipped: feDisplacementMap, the machine that moves pixels around using a second image as instructions. It's the heart of your component, and it gets the whole next lesson.

Tie-in to your mission

Open components/ui/glasscn/liquid-glass.tsx and find the <filter> block near the bottom. You can now read every line of it as an assembly line: a machine that loads an image (feImage, result="map"), a machine that consumes in2="map", and a final soft-blur machine. The only thing you can't yet explain is what the displacement machine does with that map — which is exactly lesson 0002.
